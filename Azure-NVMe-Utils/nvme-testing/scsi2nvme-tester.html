SCSI-to-NVMe Multi-Image Test Lab Builder

About SCSI-to-NVMe Multi-Image Test Lab Builder

  • Purpose: Generates an ARM template that deploys one control VM plus one VM per image in the catalog (112 images across RedHat, Debian, Canonical, SUSE, AlmaLinux, and Azure Linux), for automated SCSI-to-NVMe conversion testing.
  • Control VM: A RHEL 9 Gen2 Standard_B2s VM with Ansible, httpd, and a key server. After deployment, it automatically polls all image VMs with ping (30 forks) and runs the test playbook once all VMs are online. A live status page is served on port 80.
  • Image filters: Narrow the catalog by architecture (x64/Arm64), generation (Gen1/Gen2), disk controller (All/NVMe capable/SCSI only), or publisher. The smallest compatible VM size is auto-selected per image.
  • Disk controller “All” mode: Deploys both SCSI and NVMe variants for NVMe-capable images, producing up to 175 VMs (112 SCSI + 63 NVMe) at full catalog.
  • Force VM size: Override automatic size selection to use a specific VM size for all compatible images. The dropdown auto-filters to sizes compatible with the current image set.
  • Force images: Pick specific images from the catalog instead of deploying all filtered images.
  • Quota guard & multi-region: Set a regional vCPU limit (default 100). When projected usage exceeds the limit, VMs are automatically split across multiple regions. Multi-region deployments use a subscription-scope ARM template with per-region resource groups and hub-spoke VNet peering so the control VM can reach all image VMs via private IPs.
  • Test playbook: Paste an Ansible YAML playbook. It is gzip-compressed and base64-encoded into cloud-init custom data, then auto-executed by the control VM after all image VMs respond to ping.
  • Output: Single-region: resource-group-scope ARM template. Multi-region: subscription-scope template (deploy at subscription level, not resource group). Includes VNet/subnet/NSG, control VM with public IP, and one VM per selected image with SSH key auth. Use Copy, Copy + Portal, or Download.
  • Results: The control VM runs httpd on port 80. After the playbook completes, results are served at /results.json and an HTML validation report at /index.html, accessible via the control VM’s public FQDN.
